# Deep dive: How do React hooks really work?

[Deep dive: How do React hooks really work? | Netlify](https://www.netlify.com/blog/2019/03/11/deep-dive-how-do-react-hooks-really-work/)

[Hooks](https://ko.reactjs.org/docs/hooks-intro.html)는 본질적으로 UI에서의 부작용과 상태 동작(stateful _behavior_)을 캡슐화하는 간단한 방법입니다.
Hooks는 [React에서 처음 소개되었고](https://www.youtube.com/watch?v=dpw9EHDh2bM)
[Vue](https://css-tricks.com/what-hooks-mean-for-vue/),
[Svelte](https://twitter.com/Rich_Harris/status/1093260097558581250)과 같은 다른 프레임워크에서도 광범위하게 포함되었으며
[일반적인 JavaScript](https://github.com/getify/TNG-Hooks)에서도 도입되었습니다. 그러나 Hooks의 함수 디자인은 JavaScript의 클로저에 대한 이해가 필요합니다.

이 글에서 React Hooks를 복사하는 작업을 진행하면서 클로저에 대해 다시 소개하겠습니다. 이 글을 통해 효과적인 클로저 사용법에 대해 설명하고 어떻게 29줄의 JavaScript로 Hooks를 복사하는지 보여줄 예정입니다.
마지막으로 자연적으로 Custom Hooks이 만들어 지는 과정을 확인할 것입니다.

> _⚠️ 요약: Hooks를 이해하기 위해 사전 학습을 할 필요는 없습니다. 아래 예제를 통해 JavaScript의 기초 지식을 쌓는데 도움이 될 수도 있습니다. 걱정하지 마세요! 어렵지 않습니다!_

# 클로저란 무엇인가?

Hooks를 [사용해야 하는 이유](https://ko.reactjs.org/docs/hooks-intro.html#classes-confuse-both-people-and-machines) 중 하나는 클래스와 고차 컴포넌트(higher order component)에 대한 혼돈을 피하기 위해서입니다. 그러나 일부 사람들은 Hooks을 사용하면서 한 문제를 다른 문제로 교체했다는 느낌을 받았을지도 모릅니다. [제한된 상황에 대해](https://overreacted.io/how-are-function-components-different-from-classes/) 걱정하기보단, 먼저 [클로저에 대해](https://overreacted.io/making-setinterval-declarative-with-react-hooks/) 생각을 해야합니다. [Mark Dalgleish 가 이해하기 쉽게 요약하였습니다.](https://twitter.com/markdalgleish/status/1095025468367990784)

![](https://raw.githubusercontent.com/ESTAID/ESTAID.github.io/master/content/blog/how-do-react-hooks-really-work/images/mark-daleish.png)

클로저는 JavaScript의 기본적인 개념입니다. 그럼에도 불구하고, 새로운 개발자들에게 클로저는 많은 혼란을 주는 것으로 악명이 높습니다. Kyle Simpson의 [You Don’t Know JS](https://github.com/getify/You-Dont-Know-JS/blob/master/scope%20%26%20closures/ch5.md)에서 클로저를 아래와 같이 정의해두었습니다.

_클로저는 함수가 속한 렉시컬 스코프를 기억하여 함수가 렉시컬 스코프 밖에서 실행될 때에도 이 스코프에 접근할 수 있게 하는 기능을 뜻한다._

이는 분명히 MDN에 "함수명이 중첩될 때 파서가 해결하는 방법"으로 정의되는 렉시컬 스코프의 개념과 밀접한 관련이 있습니다.
이를 보다 잘 설명하기 위해 실제 사례를 살펴보겠습니다.

```jsx
// 예제 0
function useState(initialValue) {
  var _val = initialValue // _val useState에 의해 생성된 로컬 변수입니다.
  function state() {
    // state함수는 클로저를 사용한 내부함수입니다.
    return _val // 부모 함수에서 정의된 _val을 state()가 사용합니다.
  }
  function setState(newVal) {
    // 같다.
    _val = newVal // _val을 노출하지 않고 _val를 설정합니다.
  }
  return [state, setState] // 외부 사용을 위해 함수를 노출합니다.
}
var [foo, setFoo] = useState(0) // 배열 구조 분해 할당을 사용합니다.
console.log(foo()) // logs 0 - 최초의 initialValue
setFoo(1) // useState 스코프 안에서 _val를 설정합니다.
console.log(foo()) // logs 1 - 같은 함수를 호출하지만 새로운 initialValue를 반환합니다.
```

React `useState` hook의 기본적인 기능을 복사하였습니다. 함수 안에서 2개의 내부 함수가 있습니다. `state`와 `setState`입니다. `state`는 위에 정의된 지역 변수 `_val`를 반환하고 `setState`는 인자로 받은 파라미터(`newVal`)을 지역 변수로 설정합니다.

예제의 state의 구현이 [이상적인 getter 함수는 아니지만](https://twitter.com/sebmarkbage/status/1098809296396009472), 이는 뒷 부분에서 해결할 것입니다.
중요한 부분은 `foo`와 `setFoo`입니다. 우리는 함수 내부에 있는 `_val`을 접근하고 조작할 수 있습니다. `setFoo`는 `useState`의 스코프에 대한 권한을 유지하는데 이 참조를 클로저라고 합니다. React와 다른 프레임워크의 맥락에서 보면, 이것은 상태(state)처럼 보이는데, 바로 그런 것입니다.

만약 클로저에 대해 좀 더 깊게 공부하길 원한다면, 저는 [MDN](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Closures), [YDKJS](https://github.com/getify/You-Dont-Know-JS/blob/master/scope%20%26%20closures/ch5.md), 그리고 [DailyJS](https://medium.com/dailyjs/i-never-understood-javascript-closures-9663703368e8)를 읽기를 추천합니다. 그러나 만약 위의 간단한 코드만 이해하고 싶다면, 필요한 모든 것을 갖추었습니다.

# 함수 컴포넌트에서의 사용법

새롭게 복사한 useState를 익숙한 형태로 적용해봅시다. 우리는 `Counter` 컴포넌트를 만들 것 입니다.

```jsx
// 예제 1
function Counter() {
  const [count, setCount] = useState(0) // 위에서 정의한 useState입니다.
  return {
    click: () => setCount(count() + 1),
    render: () => console.log("render:", { count: count() }),
  }
}
const C = Counter()
C.render() // render: { count: 0 }
C.click()
C.render() // render: { count: 1 }
```

DOM을 렌더링하는 대신에, state를 `console.log`로 찍었습니다. 또한, 스크립트 안에서 동작해야 하기 때문에 이벤트 핸들러를 등록하는 대신 Counter를 위한 API 방식을 사용할 것입니다. 이 설계로 컴포넌트 렌더링과 사용자 행동에 대한 반응을 시뮬레이션할 수 있습니다.

코드가 실행되는 동안, state에 접근하기 위해 getter(함수)를 호출하는 건 실제로 `React.useState`에서 사용하는 완전한 API는 아닙니다. 이 부분을 고쳐봅시다!

# 진부한 클로저

만약 실제의 React API와 일치시키고 싶다면, state가 함수가 아닌 변수여야합니다. 만약 함수로 감싸는 것 대신에 \_val를 간단히 노출시키고 싶다면 아래와 같은 버그를 만나게 됩니다.

```jsx
// Example 0, 버그가 있는 코드!
function useState(initialValue) {
  var _val = initialValue
  // state() 함수가 없습니다.
  function setState(newVal) {
    _val = newVal
  }
  return [_val, setState] // 바로 _val를 리턴합니다.
}
var [foo, setFoo] = useState(0)
console.log(foo) // logs 0 함수 호출 없이 사용할 수 있습니다.
setFoo(1) // useState 스코프 안에서 _val를 설정합니다.
console.log(foo) // logs 0 - 헉!!
```

이것은 진부한 클로저의 문제 중 일부입니다. `useState`의 출력 값으로 `foo`를 비 구조화 할당을 했을 때, 최초의 useState에서 호출 됐기 때문에 `foo`는 `_var`를 참조합니다.
그리고 절대 바뀌지 않습니다. 이는 우리가 원하는 것이 아닙니다.
일반적으로 함수 호출 대신 변수에 불과할 뿐, _현재_ state를 반영하기 위해서 컴포넌트 state가 필요합니다. 그 두 가지 방법은 정 반대 일수 도 있습니다.

# 모듈 속 클로저

우리는 또 다른 클로저 안으로 클로저를 옮김으로써 `useState`의 수수께끼를 풀 수 있습니다.

```jsx
// Example 2
const MyReact = (function() {
  let _val // 모듈 스코프 안에서 클로저를 유지합니다.
  return {
    render(Component) {
      const Comp = Component()
      Comp.render()
      return Comp
    },
    useState(initialValue) {
      _val = _val || initialValue // 매번 실행할 때마다 새로운 값을 복사합니다.
      function setState(newVal) {
        _val = newVal
      }
      return [_val, setState]
    },
  }
})()
```

간단히 React를 복사하기 위해서 [모듈 패턴](https://addyosmani.com/resources/essentialjsdesignpatterns/book/#modulepatternjavascript)을 선택했습니다.
React와 마찬가지로 컴포넌트 state를 추적합니다. (이 예제는 오직 하나의 컴포넌트의 state 값인 `_val`를 추적합니다.)
이 디자인을 통해 `MyReact`는 올바른 클로저를 통해 매번 내부의 `_val` 값을 할당하는 함수 컴포넌트를 `render`할 수 있습니다.

```jsx
// Example 2 continued
function Counter() {
  const [count, setCount] = MyReact.useState(0)
  return {
    click: () => setCount(count + 1),
    render: () => console.log("render:", { count }),
  }
}
let App
App = MyReact.render(Counter) // render: { count: 0 }
App.click()
App = MyReact.render(Counter) // render: { count: 1 }
```

React Hooks와 좀 더 비슷한 형태를 갖추었습니다.

[모듈 패턴이나 클로저에 대해서는 YDKJS에서 좀 더 확인할 수 있습니다.](https://github.com/getify/You-Dont-Know-JS/blob/master/scope%20%26%20closures/ch5.md#modules)

# `useEffect` 복사하기

Reack Hook의 첫번째 기초인 `useState`에 대해 배웠습니다. 다음으로 중요한 hook은 [useEffect](https://ko.reactjs.org/docs/hooks-effect.html)입니다.
`setState`와 달리 `useEffect`는 비 동기적으로 실행됩니다. 다시 말해, 이는 클로저와 마주칠 기회가 더 많다는 것을 의미합니다.

지금까지 작업한 React의 작은 모델에 `useEffect`를 포함시켜 확장시킬 수 있습니다.

```jsx
// 예제 3
const MyReact = (function() {
  let _val, _deps // 이 렉시컬 스코프 안에 state값을 가지고 있습니다.
  return {
    render(Component) {
      const Comp = Component()
      Comp.render()
      return Comp
    },
    useEffect(callback, depArray) {
      const hasNoDeps = !depArray
      const hasChangedDeps = _deps
        ? !depArray.every((el, i) => el === _deps[i])
        : true
      if (hasNoDeps || hasChangedDeps) {
        callback()
        _deps = depArray
      }
    },
    useState(initialValue) {
      _val = _val || initialValue
      function setState(newVal) {
        _val = newVal
      }
      return [_val, setState]
    },
  }
})()

// usage
function Counter() {
  const [count, setCount] = MyReact.useState(0)
  MyReact.useEffect(() => {
    console.log("effect", count)
  }, [count])
  return {
    click: () => setCount(count + 1),
    noop: () => setCount(count),
    render: () => console.log("render", { count }),
  }
}
let App
App = MyReact.render(Counter)
// effect 0
// render {count: 0}
App.click()
App = MyReact.render(Counter)
// effect 1
// render {count: 1}
App.noop()
App = MyReact.render(Counter)
// // no effect run
// render {count: 1}
App.click()
App = MyReact.render(Counter)
// effect 2
// render {count: 2}
```

`useEffect`가 의존성이 바뀔 때 마다 다시 실행되는 걸 추적하기 위해, 추척하는 변수인 `_deps`를 추가로 만들었습니다.

# 마술이 아니고 단지 배열

우리는 `useState`와 `useEffect` 와 같은 좋은 함수 복사본을 가지고 있습니다.
그러나 둘 다 [싱글톤](https://en.wikipedia.org/wiki/Singleton_pattern) 형태로 제대로 구현되지 않았습니다.
마지막으로 가능한 클로저 예제를 만들기 위해 그들을 일반화해서 임의의 state와 effect 개수를 가질 필요가 있습니다.
[Rudi Yardley has written](https://medium.com/@ryardley/react-hooks-not-magic-just-arrays-cd4f1857236e)에 따르면 'React Hook은 마술이 아니고 단지 배열이다.' 라는 말이 나와있습니다.
그래서 `hooks`라는 배열을 가질 것입니다. 또한, `_val`과 `_deps` 가 절대 겹칠 일이 없기 때문에 `hooks`배열 안으로 합칠 것입니다.

```jsx
// 예제 4
const MyReact = (function() {
  let hooks = [],
    currentHook = 0 // 배열 hooks과 정수
  return {
    render(Component) {
      const Comp = Component() // run effects
      Comp.render()
      currentHook = 0 // 다음 렌더를 위해 0으로 초기화한다.
      return Comp
    },
    useEffect(callback, depArray) {
      const hasNoDeps = !depArray
      const deps = hooks[currentHook] // 타입: array | undefined
      const hasChangedDeps = deps
        ? !depArray.every((el, i) => el === deps[i])
        : true
      if (hasNoDeps || hasChangedDeps) {
        callback()
        hooks[currentHook] = depArray
      }
      currentHook++ // hook의 끝
    },
    useState(initialValue) {
      hooks[currentHook] = hooks[currentHook] || initialValue // 타입: any
      const setStateHookIndex = currentHook // setState 클로져
      const setState = newState => (hooks[setStateHookIndex] = newState)
      return [hooks[currentHook++], setState]
    },
  }
})()
```

`setStateHookIndex`는 아무것도 하지 않는 것 같지만 `currentHook` 변수에 `setState`가 갇히지 않도록 방지합니다.
만약 `setStateHookIndex`를 사용하지 않는다면, 감싸진 `currentHook`은 정의한지 오래됐기 때문에 `setState`작동이 다시 멈춥니다.

```jsx
// Example 4 속편 - 사용법
function Counter() {
  const [count, setCount] = MyReact.useState(0)
  const [text, setText] = MyReact.useState("foo") // 2번째 hook
  MyReact.useEffect(() => {
    console.log("effect", count, text)
  }, [count, text])
  return {
    click: () => setCount(count + 1),
    type: txt => setText(txt),
    noop: () => setCount(count),
    render: () => console.log("render", { count, text }),
  }
}
let App
App = MyReact.render(Counter)
// effect 0 foo
// render {count: 0, text: 'foo'}
App.click()
App = MyReact.render(Counter)
// effect 1 foo
// render {count: 1, text: 'foo'}
App.type("bar")
App = MyReact.render(Counter)
// effect 1 bar
// render {count: 1, text: 'bar'}
App.noop()
App = MyReact.render(Counter)
// // no effect run
// render {count: 1, text: 'bar'}
App.click()
App = MyReact.render(Counter)
// effect 2 bar
// render {count: 2, text: 'bar'}
```

기본적인 원리는 `hooks` 배열과 각각의 `hook`이 호출될 때 증가하는 인덱스를 가지고 컴포넌트가 렌더링될 때 재설정되는 것입니다.
또한, 자유롭게 [custom hooks](https://ko.reactjs.org/docs/hooks-custom.html)을 만들 수 있습니다.

```jsx
// Example 4, 재 방문
function Component() {
  const [text, setText] = useSplitURL("www.netlify.com")
  return {
    type: txt => setText(txt),
    render: () => console.log({ text }),
  }
}
function useSplitURL(str) {
  const [text, setText] = MyReact.useState(str)
  const masked = text.split(".")
  return [masked, setText]
}
let App
App = MyReact.render(Component)
// { text: [ 'www', 'netlify', 'com' ] }
App.type("www.reactjs.org")
App = MyReact.render(Component)
// { text: [ 'www', 'reactjs', 'org' ] }}
```

**이것은 마법이 아니라 hooks가 얼마나 중요한지 보여줍니다.**

Custom Hooks은 React인지 아니면 위에서 간단히 설계한 복사본(clone)인지에 관계없이 프레임워크에 의해 제공되는 기본 요소에서 벗어납니다.

# Hooks의 규칙을 추론합니다.

Hooks의 첫번째 규칙은 [오직 최상위에서 Hooks를 호출해야 한다는 것](https://ko.reactjs.org/docs/hooks-rules.html#only-call-hooks-at-the-top-level)입니다.
`currentHook` 변수를 사용하여 호출 순서에 대한 React의 의존성를 명시적으로 모델링했습니다. 구현을 염두에 두고 [Hook 규칙에 대한 전체적인 설명](https://ko.reactjs.org/docs/hooks-rules.html#explanation)을 읽으면 진행 중인 모든 사항을 완전히 이해할 수 있습니다.
["오직 React 함수 내에서 Hook을 호출해야 된다"](https://ko.reactjs.org/docs/hooks-rules.html#only-call-hooks-from-react-functions)라는 두 번째 규칙이 구현에 대한 필수 규칙은 아니지만, 이 규칙을 적용하여 상태 어떤 부분이 상태 관련 로직과 연관이 있는지 명확하게 구분하는 것이 좋습니다.

예를 들어 [useRef를 one-liner로 구현하거나](https://www.reddit.com/r/reactjs/comments/aufijk/useref_is_basically_usestatecurrent_initialvalue_0/), [렌더 기능을 실제 JSX를 사용하여 DOM에 마운트하거나](https://www.npmjs.com/package/vdom), 28줄 짜리 React Hook에서 생략한 100만 가지의 다른 세부 정보를 구현할 수 있습니다.
바란건대 이 글을 통해 컨텍스트에서 클로저를 사용하는 방법에 대해 경험을 쌓고, React Hooks가 어떻게 작동하는지 설명하는 유용한 모델을 얻었기를 바랍니다.

이 블로그의 초기 초안을 검토하고 귀중한 피드백을 통해 개선해 준 Dan Abramov와 Divya Sasidharan에게 감사드립니다. 남아있는 모든 에러는 제 것입니다.
